# Summary Prompt

在读完论文源码后，按这个模板写最终总结。若某节对该论文确实不适用，只做最小幅度调整。

## Goal

写一份面向 AI PhD 学生的研究级读论文笔记：既要让读者准确理解论文贡献，也要让读者知道如何复现、如何评估、还能往哪里继续做研究。

## Hard requirements

- 主体使用中文；只保留通常不必翻译、翻译后反而失真的 English 术语，例如 `Transformer`、`attention`、`ablation`、`perplexity`、`benchmark`、`optimizer`。

- 不写客套话、空泛褒奖、重复背景；结论尽量锚定到 mechanism、assumption、training recipe、experiment、ablation 或 limitation。

- 如果某条判断不是作者明确写出的，而是你根据原文推得出的，显式标记 `Inference:`。

- 只有在检查过**正文、appendix / supplement、图注、表注**之后，仍找不到某项复现关键信息时，才能写“**论文未明确给出**”。若只是部分缺失，优先写成“可确认的是……；未明确给出的是……”。不要一概写成“未明确给出”。

- 多用要点；**每条之间空一行**。关键概念、结论、限制与方向用 Markdown **加粗**。

- 如果某个章节存在 `3` 个以上并列机制、实验族或 ideas，优先使用三级标题展开，而不是把它们压进一串超长 bullet。

- `## 4. 方法` 默认按 `### 4.1 / 4.2 / ...` 拆开；每个 `4.x` 子节具体讲什么，要由论文自己的方法结构决定；即使只有一个主方法块，也至少保留 `### 4.1`，并在该子节下继续用 bullet 分点描述。

- 如果一个 bullet 同时包含“研究问题 + 设置 + 结果 + 数字 + 解释”，优先拆成 `2-4` 条或改成小节，保持可检查性。

- 把下面的结构要求当成**输出格式 contract**，不是软建议；如果草稿没有满足，先重写结构再交付。

- 如果 `./figures/header.png` 存在，标题和元信息之后必须插入头图。

- 如果某一部分明显需要图来说明，例如方法结构、主结果、复杂度曲线、attention 可视化，就插入对应图片，并补一条简短图注，说明“为什么看这张图”。

- 如果 `2-3` 张图共同说明同一个点，可以放进同一个居中的 HTML 图片块；图注要明确说明左图 / 右图 / 各子图分别看什么，以及合起来说明什么。

- **每个单独展开的实验**后面都要紧跟它自己的图或表；不要把图表统一堆在实验节最后。

- 图片优先用 `./figures/` 中已经物化好的文件，统一使用居中的 HTML 包裹，例如 `<p align="center"><img ... /></p>`；通常 `68%` 到 `78%`，较高或较窄的图用 `42%` 到 `60%`。若 `manifest.json` 提供了 `suggested_width_pct`，优先采用。

- **任何出现的 `<img>` 都必须被 `<p align="center">...</p>` 完整包裹**；裸 `<img>` 视为格式错误。

- 如果 `./reference_links.json` 里有 prior work 的链接，优先写成 `[方法名](链接)`。

- **不要**把整个 Markdown 超链接包进反引号；写 `[Transformer-XL](...)`，不要写 `` `[Transformer-XL](...)` ``。

- 如果论文有重要表格，尤其是主结果表、ablation 表、benchmark 总表，应转写为 Markdown 表格。若原表过宽，可只保留关键行列，并明确写成“**表格节选**”。

- 只要保留实验与评估这一节，就默认从该节对应的 `### x.1` 开始组织；若训练方法被删去并顺延编号，就相应写成 `### 5.1`、`### 5.2` ...；即使只有一个主要实验族，也不要退化成一串大 bullet。

- 每个实验子节都必须显式出现这三个标签：`- **研究问题**：`、`- **设置**：`、`- **结果**：`。

- `## x. 洞见与创新` 必须拆成当前节编号对应的 `### x.1 Insight` 与 `### x.2 创新点`；格式示例 `（格式：**【创新点解决的问题】 -> 【受哪个 insight 启发】 -> 【设计了什么创新点】**）` 必须单独成行，不要写成 list item。

- 每个 idea 都必须写成 `### Idea 1：...`、`### Idea 2：...` 这种三级标题，并显式出现这五个标签：`- **要解决的问题**：`、`- **继承的 insight / 机制 / 观察**：`、`- **方法设计**：`、`- **最小可行实验**：`、`- **主要风险**：`。

- `## 8. 潜在问题与后续方向` 不要写成一串混合 bullet；优先拆成 `### 8.1 潜在问题` 与 `### 8.2 后续方向`。其中 `8.1` 要区分“作者明确承认的局限”与“基于论文证据可推导的潜在问题”。

- `## 8. 潜在问题与后续方向` 与 `## 9. AI idea brainstorming` 要明显区分：`第 8 节` 更贴近论文现有方法、证据与可直接延伸的问题；`第 9 节` 则应提出更主动、更发散、更不受限、具备独立 research identity 的想法，不要只是把 `8.2` 改写一遍。

- 完成初稿后，必须再做一轮**结构归一化检查**：把平铺方法改写成 `### 4.1 / 4.2 / ...`，把平铺实验改写成与当前实验节编号一致的 `### x.1 / x.2 / ...`，把 `洞见与创新` 改写成 `### x.1 Insight / x.2 创新点`，把平铺 idea 改写成 `### Idea X`，把所有裸 `<img>` 改成居中包裹。

## Output template

```markdown
# <paper title>

- **arXiv**：<arxiv id>

- **一句话核心结论**：<最核心 claim>

<若有头图，在这里插入居中的 HTML 图片标签，例如 `<p align="center"><img ... /></p>`>

*图注：一句话说明它为何最能代表全文。*

## 1. 问题定义

- 形式化说明论文要解决什么问题；交代输入、输出、目标、约束，以及真正被优化的核心对象。

## 2. 核心挑战

- 直接点名 2-4 个具体 prior methods / baselines / paradigms / theories，说明它们为何不足；不要写“许多已有方法”“另一类方法”“一些工作”。

- 难点来自 computation、data、optimization、credit assignment、generalization、evaluation、system constraints、identification、deployment 或别的结构性限制，要说清楚。

## 3. 动机

- 用递进的问题链解释作者为什么会想到这个方案；写成第一性原理推导，而不是复述 introduction。

## 4. 方法

### 4.1 <方法点 1>

- <在这一小节下，用 bullet 分点整理与该方法点直接相关的内容。>

### 4.2 <方法点 2>

- <继续按论文自己的方法结构展开；这一小节具体讲什么由文章内容决定。>

### 4.3 <方法点 3>

- <若还有更多方法点，就继续扩展为 `### 4.4`、`### 4.5` ...；保留到论文实际需要的小节数即可。>

<若某个点更适合用一张方法图说明，就插入单图与图注；若两三张图共同说明同一个点，可放进同一个居中的 HTML 图片块，并在图注里写明左图 / 右图各看什么、合起来说明什么。>

## 5. 训练方法

- 只有当论文确实训练了模型时保留这一节；否则删去并顺延编号。

- **模型与初始化**：<底座 / 参数规模 / 初始化 / 关键结构配置>

- **优化与日程**：<optimizer / lr schedule / warmup / step / regularization / 关键损失>

- **数据与样本构造**：<训练数据、过滤、配比、采样或标注方式>

- **训练流程**：<阶段划分、训练接口、样本流转、真正决定结果的流程安排；是否存在特殊设置由论文自己显现出来，不要生硬罗列概念名>

- **计算与实现**：<batch / hardware / 精度 / 吞吐或成本信息；若缺失则写清缺失项>

- **复现缺口**：<已确认的缺失信息；若没有可省略>

## 6. 实验与评估

### 6.1 <实验族名称>

- **研究问题**：<这一组实验到底要回答什么问题>

- **设置**：<和谁比、在哪个数据 / benchmark 上比、metric 是什么、协议是什么>

- **结果**：<最重要的现象、数字、结论、边界>

<紧跟该实验的小图/小表；必须用居中的 HTML 图片或 Markdown 表格>

*图注 / 表注：一句话说明为什么看这张图或这张表。*

### 6.2 <实验族名称>

- **研究问题**：...

- **设置**：...

- **结果**：...

<继续按这个模式展开后续实验族；如果只有一类实验，也保留 `### 6.1` 这种写法。>

## 7. 洞见与创新

### 7.1 Insight

- **Insight 1**：...

- **Insight 2**：...

### 7.2 创新点

（格式：**【创新点解决的问题】 -> 【受哪个 insight 启发】 -> 【设计了什么创新点】**）

- **【创新点解决的问题】 -> 【受哪个 insight 启发】 -> 【设计了什么创新点】**

- **【创新点解决的问题】 -> 【受哪个 insight 启发】 -> 【设计了什么创新点】**

## 8. 潜在问题与后续方向

### 8.1 潜在问题

#### 作者明确承认的局限

- 优先整理作者在正文、appendix / supplement、discussion、limitations、图注或表注中**明确承认**的局限、假设、失败边界或适用范围限制。

- 如果作者没有集中写 limitation，不要编造；可以写“**作者未集中列出 limitation，但文中可直接确认的约束包括……**”。

#### 基于论文证据可推导的潜在问题

- 基于方法假设、实验覆盖范围、ablation 缺口、失败案例、图表现象或结果反常点，补充你认为最值得警惕的潜在问题。

- 尽量说清楚：你是从**哪一类证据**推出来的、它**可能在什么条件下失效**、以及它**会影响论文的哪部分结论**。

- 不要只写“可能泛化不好”“可能不稳定”这类空泛判断；尽量把问题落到具体机制、数据条件、评测范围、系统约束或复现风险上。

### 8.2 后续方向

- 把最值得继续深挖的后续方向单独展开，不要压成一句“值得继续推进的方向包括……”。

- 优先写那些**直接回应 8.1 中问题**、或**顺着本文 insight / 方法 / 实验现象自然延伸**出来的方向。

- 对每个方向，至少交代：**想解决什么问题**、**为什么值得做**、以及**最小可以怎样验证它是否有效**。

- 这一节偏“贴着本文继续往前走”的研究延伸；更发散、更大胆的想法放到 `## 9. AI idea brainstorming`。

## 9. AI idea brainstorming

- 这一节要明显区别于 `## 8. 潜在问题与后续方向`：不要只把第 8 节里的 follow-up 重写成更长的 bullet，而要提出**更主动、更有新意、具备独立 research identity** 的想法。

- **Brainstorming** 应当强调**发散、不受限、敢于跳出论文原始设定**：可以跨任务、跨模态、跨训练范式、跨系统接口，或重新定义问题设定；不要被论文当前实验边界过度束缚。

- 可以继承本文的 insight、机制、实验现象或 limitation，但最终提出的 idea 应当包含**新的方法设计、训练/评测接口改动、任务设定变化，或跨领域迁移**，而不只是“把某个问题补做完整”。

- 优先写那些**如果单独拿出来，也能形成一个清晰 research project 雏形**的 idea；避免只写“扩大规模试试”“换个数据集试试”“补一个 ablation”这类过近、过弱的延伸。

### Idea 1：<标题>

- **要解决的问题**：<具体问题，不要写大而泛的愿景句>

- **继承的 insight / 机制 / 观察**：<明确点名来自本文哪里>

- **方法设计**：<准备怎么改方法 / objective / pipeline / system / evaluation；要体现出比第 8 节更主动的设计性>

- **最小可行实验**：<数据、benchmark、baseline、metric>

- **主要风险**：<最可能失败的地方>

### Idea 2：<标题>

- **要解决的问题**：...

- **继承的 insight / 机制 / 观察**：...

- **方法设计**：...

- **最小可行实验**：...

- **主要风险**：...

<继续按这个模式写到 3-5 个 ideas。>
```

## Quality bar

最终结果应帮助读者快速判断：

1. 论文真正贡献是什么；

2. 贡献依赖什么 insight；

3. 复现时要抓哪些训练与评测细节；

4. 它具体绕开、重组或超越了哪些 prior methods；

5. 方法最脆弱的地方在哪里；

6. 哪些延伸方向最值得继续做。
